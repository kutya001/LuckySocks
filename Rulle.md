Главное правило GitHub Pages: **он хостит только статические сайты** (HTML, CSS, JavaScript, изображения, шрифты). Серверные языки (PHP, Python, Node.js, Ruby) и базы данных на нем напрямую не работают.

Однако это не значит, что вы ограничены простым HTML. Вы можете использовать любые современные технологии для *сборки* статического сайта, а затем деплоить результат.

Вот подробный обзор технологий, которые можно использовать, разделенный по категориям :

### 1. Генераторы статических сайтов (SSG)
Это самый популярный выбор. Вы пишете контент в Markdown или компонентах, а SSG генерирует готовые HTML-файлы.
*   **Jekyll:** Единственный SSG, который GitHub Pages поддерживает **нативно** (из коробки). Если вы пушите исходники Jekyll в ветку `main`, GitHub сам соберет сайт.
*   **Astro:** Современный и очень быстрый фреймворк. Позволяет использовать компоненты из React, Vue, Svelte, но на выходе отдает чистый статический HTML (отлично для SEO и скорости).
*   **Hugo:** Написан на Go, собирается молниеносно. Идеален для больших блогов и документации.
*   **Eleventy (11ty):** Простой и гибкий генератор на JavaScript. Не навязывает никаких фреймворков.
*   **Gatsby:** Мощный SSG на React с богатой экосистемой плагинов (хотя сейчас уступает место Astro по части чистого статического вывода).

### 2. Фронтенд-фреймворки (SPA - Single Page Applications)
Вы можете создать полноценное приложение на **React, Vue, Angular, Svelte** или **SolidJS** и собрать его в статические файлы (`npm run build`).
*   *Важный нюанс:* Поскольку на GitHub Pages нет сервера, который мог бы перенаправлять все запросы на `index.html`, **клиентский роутинг (React Router, Vue Router) будет ломаться** при обновлении страницы (будет выдавать 404).
*   *Решение:* Использовать Hash-роутинг (адреса вида `site.com/#/about`) или настраивать кастомный `404.html`, который будет перенаправлять на `index.html` (хак, который часто используют).

### 3. Инструменты для создания документации
Если ваша цель — документация, Wiki или технический блог, используйте специализированные инструменты:
*   **VitePress / VuePress:** Отличный выбор для документации, работает на Vue и Vite.
*   **Docusaurus:** Очень популярный генератор документации от Meta на базе React.
*   **MkDocs (с темой Material):** Пишете в Markdown, собираете через Python. Выглядит очень профессионально.
*   **Docsify:** Рендерит Markdown прямо в браузере (не требует этапа сборки, просто загружаете `.md` файлы).

### 4. Сборщики (если пишете с нуля)
Если вы не используете фреймворки, а пишете на чистом JS/TS, CSS (SCSS/Sass/PostCSS), вам нужны сборщики:
*   **Vite:** Современный стандарт. Быстрый, простой в настройке.
*   **Webpack / Parcel / Rollup:** Классические инструменты для бандлинга JS и CSS.

### 5. Технологии для деплоя (CI/CD)
Как физически доставить код на GitHub Pages?
*   **GitHub Actions (Рекомендуется):** Встроенная CI/CD система. Вы создаете `.yml` файл в папке `.github/workflows/`, который при каждом `git push` автоматически собирает ваш сайт (например, через `npm run build`) и пушит результат в ветку `gh-pages` или в специальную среду GitHub Pages.
    *   *Популярные экшены:* `actions/deploy-pages`, `peaceiris/actions-gh-pages`.
*   **Пакет `gh-pages` (npm):** Утилита для Node.js. Добавляется в `package.json` как скрипт (`"deploy": "gh-pages -d dist"`). При запуске она сама создает ветку `gh-pages` и пушит туда папку со сборкой.
*   **Прямой пуш:** Вручную коммитить собранные файлы в ветку `main` или `gh-pages` (не рекомендуется, так как засоряет историю репозитория скомпилированным кодом).

### 6. Как добавить "динамику" (Бэкенд для статического сайта)
Поскольку своего бэкенда на GitHub Pages нет, для форм, баз данных и авторизации используют BaaS (Backend-as-a-Service) и Serverless:
*   **Базы данных и Авторизация:** Firebase, Supabase, Appwrite.
*   **Обработка форм:** Formspree, Web3Forms, Getform (позволяют отправлять данные с HTML-форм на email без бэкенда).
*   **Поиск:** Algolia, Pagefind (работает локально в браузере), Lunr.js.
*   **Комментарии:** Giscus (на базе GitHub Discussions), Utterances (на базе GitHub Issues).
